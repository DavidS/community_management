PDK Planning 54 225 glennsarti (wip) PDK Templating v2.0 I am a bit against reworking the template system. Not because I think it doesn't require rework, but currently when I tell my clients about pdk-templates I just say it works like modulesync. Which is well documented and used by many. While, the current implementation isn't perfect, making the user switch to another system is going to piss them off. Many users coming from the modulesync workflow will start using pdksync with PDK. So for a large corporation to convert to PDK and forgetting everything about how modulesync is a hard pill to swallow. Regarding the opt-in strategy. This seems ideal. I like it. The way I use pdk-templates is always the same: 1. Mirror official external templates into internal git repository 2. Fork the internal mirror into a namespace that is intended for heavy internal usage 3. Make changes that are reflective to my environment 4. Remove all calls to the internet (rubygems.org, puppet.com, github.com) 5. Add all my exclusions to the template_configs 6. Create branches for each tag of the upstream templates With this in mind there should be an air-gaped setting too. Provide a way to fill in special environment variables in the templates.yml that are then used in the template rendering. ```yaml pdk_templates: config: air_gaped: true GEM_SOURCE: "ENV['GEM_SOURCE']" ci_system: - travis - appveyor scm_system: github_enterprise upstream_template_repo: github.com/puppetlabs/pdk-templates internal_template_repo: github.com/nwops/pdk-templates ``` On a side note. What I would like to see is the templates.yml file become the configuration code for describing an environment. Take the upstream pdk-templates from puppet and "merge" them with an internal (subset). Whereby, the internal subset files take precedence over the upstream version. This should be similar in behavior to `Hash.merge.` What this might look like: 1. pdk new module --template_url=https://github.com/nwops/pdk-templates.git 2. pdk clones https://github.com/nwops/pdk-templates.git 3. pdk references the templates.yml file and finds the upstream_template_repo 4. pdk clones the upstream template repo 5. pdk starts to create new files and looks for template in internal_template_repo first and fails back to upstream templates for the requested template. This keeps the user from having to keep an extra fork of pdk-templates around and constantly maintain them with upstream changes. logicminds 213
